Отчет о выполнении практической работы: Дополнительные возможности Docker
В рамках данной практической работы были изучены продвинутые механизмы управления жизненным циклом контейнеров, мониторинга их состояния и оптимизации потребления системных ресурсов. Практика была направлена на углубленное понимание работы Docker в условиях, приближенных к production.

Выполненные задания
1. Вывод логов в файл
Освоен механизм сохранения стандартных потоков вывода контейнера (stdout и stderr) в файловую систему хоста.

Выполнено перенаправление вывода команды docker logs в текстовый файл для дальнейшего анализа.
<img width="761" height="509" alt="image" src="https://github.com/user-attachments/assets/f9082769-7f74-4d1f-9db7-5f8a3ea76b53" />

Данный подход позволяет проводить постмортем-анализ и диагностику даже после остановки или удаления контейнера.

2. Мониторинг ресурсов (Docker Stats)
Проведен анализ потребления системных ресурсов контейнерами в реальном времени с помощью утилиты docker stats.

Изучены метрики использования CPU, оперативной памяти и сетевого ввода-вывода.
<img width="1167" height="623" alt="image" src="https://github.com/user-attachments/assets/498a7fd8-13fe-438d-8348-2280f3d7ee06" />

Реализовано создание "мгновенных снимков" (snapshots) статистики путем сохранения вывода в файл без потокового обновления. Это необходимо для фиксации состояния системы под нагрузкой.

3. Ограничение ресурсов
На практике применен механизм жесткого ограничения вычислительных мощностей контейнера для предотвращения монополизации ресурсов хоста.

Установлены лимиты на оперативную память (--memory) и процессорное время (--cpus).

Проверено динамическое изменение лимитов на уже запущенном контейнере с помощью команды docker update.
<img width="976" height="515" alt="image" src="https://github.com/user-attachments/assets/340b5d56-4d98-40e6-830b-13871b05edf4" />

Данная настройка критически важна для стабильности многоконтейнерных сред.

4. Экспорт контейнера в архив
Выполнено создание полной резервной копии файловой системы контейнера.

С помощью команды docker export состояние работающего контейнера было сохранено в переносимый tar-архив.
<img width="1017" height="547" alt="image" src="https://github.com/user-attachments/assets/1f704c86-71cb-4213-9949-5d3b3c0fa33d" />

Данный метод позволяет сохранять все изменения, произведенные внутри контейнера с момента его запуска, и используется для создания бэкапов или передачи окружения.

5. Импорт контейнера из архива
Отработан процесс восстановления рабочего окружения из ранее созданного архива.

Использована команда docker import для создания нового Docker-образа на основе tar-архива.
<img width="1101" height="698" alt="image" src="https://github.com/user-attachments/assets/cff1d84b-cdaa-42a2-aaed-63ad285d2e0a" />
<img width="841" height="561" alt="image" src="https://github.com/user-attachments/assets/fb03ca9e-ef1f-4d03-a0af-66e76969cdcb" />

Из восстановленного образа успешно запущен новый контейнер, полностью идентичный исходному по состоянию файловой системы.

Итог
В ходе работы закреплены навыки администрирования Docker: от диагностики и мониторинга до управления ресурсами и миграции приложений. Все операции выполнялись через CLI, что подтверждает готовность к работе с контейнерами в серверном окружении.
